ICS Sentinel

An AI SOC analyst for industrial networks

Detects attacks on unauthenticated Modbus TCP control traffic, then uses Claude to triage every alert into plain English, MITRE ATT&CK for ICS context, and concrete response actions.

Modbus has no authentication and no encryption — any device on the network can command a PLC. The protocol will obey whoever speaks it first.

01 · The pipeline

Five stages. One narrowing funnel.

Every number below is from a real default run (python -m ics_sentinel.demo, seed 42, 60-second window) and verified by tests. Monitoring is fully passive — Sentinel reads a SPAN/TAP-style mirror of the traffic and never injects a packet into the process.

02 · The plant & the threats

A water-treatment skid, watched end to end.

Tank levels follow real pump physics (hysteresis control), so the baseline is physically coherent — not random noise. Five attack scenarios, five rules that catch them.

authorized traffic (HMI polls, EWS writes) attacker traffic trust boundary

Unauthorized write

Normal-looking commands… from the wrong host.

T0855

Write-source allowlist

Only 10.0.0.11 may write. Anyone else trips the wire.

Dangerous setpoint

Pump setpoint forced to 200% — from the authorized workstation (insider / compromised).

T0836T0831

Safe-range check

Setpoint 40002 must stay within 20–90. Value, not just source.

Recon scan

160 register/unit probes in ~4 seconds (benign polling touches 6 points).

T0846

Distinct-points window

More than 12 distinct points in one window means someone is mapping the plant.

Malformed frames

Illegal function codes (0x5A, 0x00), corrupt payloads.

T0814

Structural validation

Every frame parsed against the Modbus spec. Garbage never reaches the rules behind it.

Replay flood

One captured command replayed 40×/second, byte-identical.
